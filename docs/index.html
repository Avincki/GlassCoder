01

The mental model

GlassCoder is the harness, not the model. All inference happens in an external server you run yourself — vLLM, SGLang, llama.cpp, Ollama, or a hosted API. GlassCoder talks to it over one boundary, the seam: POST /v1/chat/completions. Which server answers is configuration, never architecture.

Everything above that seam is a small loop you can read:

Observe  →  Think  →  Act  →  Result
   ↻ repeat until the goal is met and verification passes,
     or a step / token / time / cost limit trips

Three things are treated as first-class, and they are why the application exists at all: every step is logged so a run can be replayed; every change is visible as a diff before it lands; every run is measured so two configurations can be compared.

Two ways to drive it

glasscoder
The console host. Non-interactive, meaningful exit codes, meant for CI and for running the task suite.
GlassCoder.Wpf
The desktop app. Same services, same configuration — adds a live transcript, diff review with approval, and the metrics dashboard.

They share one bootstrap, so anything you measure in one applies to the other.

02

Getting a first run

You will need

  • The .NET 10 SDK.
  • A model server answering on an OpenAI-compatible /v1 endpoint, serving at least one alias.
  • Docker, if you want the agent to build or run tests. Without it those two tools refuse rather than running on your machine — see defaults that refuse.

  1. Build it

    dotnet build
dotnet test        # optional: 195 tests, ~25s

  2. Point it at your server

    Open config/appsettings.json and set the worker role's endpoint and alias. The alias is the name your server serves — never a checkpoint path. GlassCoder rejects paths at startup, because a path silently pins an experiment to one machine.

    "worker": {
  "Endpoint": "http://localhost:8001/v1",
  "ModelAlias": "worker",
  "Temperature": 0.2
}

  3. Check what the agent can see

    This resolves the whole service graph and prints the tools in the order the model is offered them. If configuration is wrong, you find out here rather than mid-run.

    glasscoder tools

  4. Run a read-only goal

    Start where the agent cannot break anything. Nothing is writable until you say so, so this is safe by construction.

    glasscoder run --repo C:\src\your-project \
  --goal "Find where request retries are implemented and explain the backoff."

  5. Read what happened

    Two files appear under the working directory. The .log is for you; the .jsonl is the full transcript, and it is the one that matters.

    logs/glasscoder-20260722.log      # human
logs/glasscoder-20260722.jsonl   # every step, replayable
metrics/metrics.jsonl            # one line per run
Watch this first

On a new model, the number that tells you whether anything else is worth trying is tool-call validity rate. If the model cannot reliably emit a well-formed tool call, no amount of prompting will fix the runs above it. Constrained decoding is on by default precisely to make that rate high.

03

Configuration

Everything lives in config/appsettings.json. No endpoint, alias, budget or limit is hardcoded anywhere. That is what makes an experiment reproducible: an arm is a config file, never a code change.

The sections

SectionControlsReach for it when
Models Endpoints, aliases, sampling, cost per million tokens, constrained decoding — per role (worker, drafter, critic). Swapping models or serving several at once.
Workspace Repo root, readable paths, writable paths, denied globs, symlink policy. Letting the agent edit anything at all.
Agent Step, token, wall-clock and cost limits; the system prompt. Runs are too long, too expensive, or give up too early.
Context Window budget, compaction threshold, the lean always-loaded root files. Long runs; A/B testing whether context earns its tokens.
Verification Diagnostic cap, pre-write compile check, whether a breaking edit is refused. Edits land broken, or the check costs too much latency.
Sandbox Docker vs host, image, network policy, hardening, bash opt-in. Build and test tools refuse to run.
Approval Whether a human must approve each write, and the timeout. You want to review diffs before they land.
Critique Multi-critic refutation: role, critic count, whether it gates. Phase 2 — watching recovery rate.
Orchestration Sub-agents and parallel fan-out. Phase 5, and only after the solo loop is good.
Provenance Commit stamping and context-freshness judgement. Comparing pass@1 on fresh versus stale context.
Logging
Metrics
Telemetry		 Sink locations, the content-redaction switch, OTLP endpoint. Sending traces somewhere, or logging without source content.

Overriding without editing the file

Environment variables win over the file, and command-line arguments win over those:

# double underscore is the section separator
set GlassCoder__Agent__MaxSteps=50
set GlassCoder__Logging__LogSourceContent=false

For a whole set of overrides, layer a second file instead — this is how arms work, and two are shipped: 

glasscoder run --config config/phase0.json --goal "..."
glasscoder run --config config/phase1.json --goal "..."
04

Defaults that refuse

Four defaults will stop the agent doing something you might have expected it to do. Each is deliberate, and each fails closed — the safe direction — so you should expect to meet them and turn them off knowingly.

Nothing is writable

Workspace:WritablePaths ships empty, and empty means no file may be written at all. A harness that cannot write is harmless; one that writes wherever it likes is not. Opt in explicitly: 

"WritablePaths": [ "src", "tests" ]
A build is arbitrary code execution

build and run_tests run inside a container with your repo mounted and the network dropped, except for a restore. If Docker is not reachable they return sandbox_unavailable rather than quietly running on your machine. To accept that risk deliberately, set Sandbox:AllowUnsandboxedExecution to true.

Approval fails closed

If Approval:RequireApprovalForWrites is on and nothing can ask a human — the console host, for instance — writes are refused, not allowed. A timeout counts as a refusal too. Approving by default would turn a safety setting into a no-op exactly when someone believed it was protecting them. Use the desktop app when you want this on.

bash is off

The bash tool is exactly as privileged as running a build, so it is opt-in via Sandbox:EnableBashTool and only available inside the container. Prefer the dedicated tools: they give the agent better observations and you better logs.

05

Tools & the verification ladder

Eight tools are offered, in this order. Order is part of the contract — the model reaches for what it is shown first, so the cheap, high-value oracles come before the expensive ones.

ToolDoesNotes
update_todosRecords the agent's plan.Whole list each time; snapshotted into every step.
read_fileReads a line range.Truncates and says so.
grepRegex search across the workspace.The cheapest way to locate code.
globLists paths matching a pattern.Skips .git, bin, obj.
create_fileWrites a new file.Refuses to overwrite — creation and modification stay separate verbs.
edit_fileReplaces an exact, unique string.Errors if the target is missing or ambiguous.
buildParsed dotnet build.The authoritative compile gate.
run_testsParsed dotnet test.Reports failing test names, not just counts.

Every tool returns a single observation object, and errors are observations, not exceptions. A tool failure is information the agent acts on; it never ends the run.

The ladder, cheapest oracle first

After an edit, verification climbs from the bottom. Each rung runs only if the one below it passed, so tests never run on code that does not compile.

06 CritiqueMulti-critic refutation. Off by default; reports rather than gates. optional
05 Full suiteBefore a change is accepted. gates
04 Unit testsOnce it compiles. Minutes. gates
03 AnalyzersConvention drift and nullability. reports only
02 CompileHallucinated APIs, wrong signatures. Seconds. gates
01 SyntaxAfter every edit, in process. Milliseconds. gates

Two rungs also run before a write reaches disk: edit_file parses the result and compiles the project in memory, and refuses the edit if it would introduce a new error. Errors that were already there never block an edit — the agent is usually editing precisely because something is broken.

Why you never see a wall of errors

Diagnostics are summarised before the model ever sees them: first error per file, deduplicated by code, capped at ten, sorted by position, with the true total always reported. One bad edit can produce two hundred errors that are all the same error — handing those over raw would consume the entire context window and bury the cause.

Exit codes

Distinct on purpose: CI has to be able to tell "the agent could not solve it" from "this is misconfigured", because only one of those is worth retrying.

0
Success — the run finished and, where an oracle existed, it passed.
1
The task did not pass.
2
Bad configuration or command line. Retrying will not help.
3
A step, token, time or cost limit stopped the run.
4
The model endpoint could not be reached, or failed the call.
5
Cancelled.
70
Something went wrong inside the harness itself.
07

What a run leaves behind

The transcript

logs/glasscoder-DATE.jsonl holds one JSON object per event. For every loop iteration it records the full prompt as sent, the response, each tool call with its arguments, whether the call parsed, the result, token counts, per-call latency, the context size, and the outcome. A closing record holds the goal, the system prompt and how the run ended.

That is enough to reconstruct the run completely — which is not a claim, it is a test: the harness reads its own log back into a transcript, and a truncated final line (what a crash leaves) costs you that line and nothing else.

Redaction

Secrets are never logged: secret-named properties are blanked and secret-shaped values are scrubbed, in both sinks. To drop source content entirely while keeping the step skeleton — indexes, tool names, statuses, tokens, latencies — set Logging:LogSourceContent to false.

The metrics

metrics/metrics.jsonl holds one object per run, with derived rates included so no reader has to reimplement a definition. Load it into a notebook directly.

IndicatorMeans
passAtOneThe oracle went green. Null until something graded the run — the loop cannot know, only a task suite can.
toolCallValidityRateFraction of tool calls that parsed and executed. The best early diagnostic for a weak model.
steps · totalTokensSteps-to-solve and tokens-to-solve.
wallClockMsThe local cost function. Always read next to pass@1.
compileErrorRatePerEditSharpest cheap quality signal.
editsToGreenEdits taken to restore a compiling state after a break.
recoveryRateDid the agent get itself back out of a failing state.
cascadeRatioErrors reported against root causes shown. Validates the summariser.
costUsdFrom the per-role token prices you configure.
contextFreshWhether the always-loaded context was at least as new as the code.
A deliberate null

An ungraded run records passAtOne: null, not false. Recording false would quietly poison every average taken over those rows. If you want real pass@1 numbers, run the suite.

08

Measuring: the suite and the arms

The task suite

Eight tasks, ordered by the skill they stress, each with a passing test as ground truth. Every fixture is a self-contained project whose exit code is the verdict — no human grading, no test framework to restore, and it runs identically inside a network-dropped container. 

glasscoder suite                              # all eight
glasscoder suite --suite-task suite-03-off-by-one
#TaskStresses
1Make one failing test passNavigation, local reasoning
2Implement a documented but empty functionReasoning from a spec
3Find and fix an off-by-one bugDebugging
4Thread a parameter through a call chainMulti-file editing
5Refactor with behaviour unchangedRestraint
6Wire a module into the buildBuild knowledge
7Add a feature across three filesLong-horizon planning
8Reproduce a bug from a report, then fix itComprehension

Run glasscoder fixtures to check the suite itself. It reports whether each fixture starts in the state its task assumes — seven start red and task 5 starts green, because its oracle is precisely stays green. A fixture in the wrong starting state makes every pass@1 computed from it meaningless.

Ablation arms

An arm is a set of configuration overrides and nothing else. Each standard arm moves exactly one lever, because an arm that moves two measures neither. 

glasscoder ablate --work ./ablation
baseline
Everything on. The number the others are read against.
no-constrained-decoding
Does constrained decoding actually buy tool-call validity?
no-prewrite-verification
Is the in-memory compile check worth its latency?
no-context
Does the always-loaded root earn its tokens?
unsummarised-diagnostics
Does the summariser change outcomes, or only comfort?

Results land in the same metrics file, tagged with the arm, and the run prints a comparison table. Add your own arms by naming any configuration key.

09

The desktop app

Three surfaces, which are the three first-class requirements made visible rather than an arbitrary choice of screens. Type a goal in the bar, press Run, and the transcript fills as the loop turns.

Transcript

Every step as it happens, with tools, outcome, tokens and latency. Filter by tool, by severity, from a step number, or search across prompts and results. Selecting a step shows the full prompt, response and tool results underneath.

Changes

Every change the agent proposed, as a before/after diff with its file, line range and status — Proposed Applied Rejected — and the compile or test result that change produced, attached to it. Refused changes appear too: a view that only showed successful writes could not tell you what the agent tried to do.

Turn on Approval:RequireApprovalForWrites and each write pauses here with Approve and Reject next to the actual diff, so you decide while looking at the change rather than at a filename in a dialog.

Metrics

Reads the same JSONL a notebook would, grouped by arm, task or source. This is where two arms get compared: pass@1, tool-call validity, tokens-to-solve, wall-clock, compile-error rate, recovery, cascade ratio and cost, side by side.

Working through the phases

The intended way to use GlassCoder is one variable at a time. Do not advance until the current phase is instrumented and green on a couple of tasks — otherwise you are changing two things and learning nothing.

0Bare instrumented loopOne worker, read/grep/glob, full logging, constrained decoding. No editing.watch: tool-call validity
1Close the loopAdd edit, build and run_tests, with the diagnostic summariser from day one.watch: compile-error rate per edit, edits-to-green
2VerificationTest gate plus the critique pass, ideally on a different model family.watch: recovery rate
3ContextPlug in the lean root context and A/B it against none.watch: pass@1, tokens-to-solve
4Scale the modelPoint the same harness at a larger drafter, or a hosted arm. Config only.watch: pass@1 against wall-clock
5OrchestrateSub-agents, parallel fan-out, multi-critic voting. Last on purpose.watch: quality delta versus solo
6FreshnessHeadless CI, provenance stamping, hardened sandbox.watch: pass@1 fresh versus stale context

Phases 0 and 1 ship as config files. Start here: 

glasscoder suite --config config/phase0.json   # read-only baseline
glasscoder suite --config config/phase1.json   # edit + verify
11

When something refuses

You seeWhat it means, and what to do
sandbox_unavailable Docker is not reachable and running repository code on the host is not permitted. Start Docker, or set Sandbox:AllowUnsandboxedExecution to true to accept the risk knowingly.
path_not_allowed The path is outside the readable or writable set, or excluded by a denied glob. For writes this usually means WritablePaths is still empty.
approval_refused A human rejected the change, the approval timed out, or approval was required with nothing able to ask. Nothing was written.
verification_failed The edit would have introduced a new compile error, so it was refused before reaching disk. The summarised diagnostics come back with it.
ambiguous_target edit_file found the target text more than once. This is a guardrail, not a bug: an edit that could land in two places would land in the wrong one silently.
exit 2 at startup Configuration failed validation. The commonest cause is a ModelAlias that looks like a checkpoint path — use the alias your server serves.
exit 3 A limit tripped. Raise Agent:MaxSteps, MaxTotalTokens, MaxWallClockSeconds or MaxCostUsd — or ask a smaller question.
exit 4 The endpoint failed. Check the server is up, the port matches, and the alias is one it actually serves.
contextFresh: false A source file is newer than your always-loaded context. The stale reason names the file. The harness's own output — logs, metrics — is excluded, so a run cannot invalidate the next one.
Two things are unproven

The Docker executor's request construction is tested but its container lifecycle has never met a real daemon — the first Phase 1 run is where that gets exercised. And no run has yet touched a real model: everything is verified against fakes and a socket-level stub, so the machinery is right but the first honest pass@1 comes from your own server.
