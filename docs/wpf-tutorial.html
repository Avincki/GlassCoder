About the mockups

The window images below are reproductions, not screenshots — colours, columns and wording are taken from MainWindow.xaml, the three views and DiffConverters.cs. The numbers in them are illustrative: your model, your machine and your endpoint will produce different token counts and latencies. 00

A repo that is broken on purpose

Pointing the agent at your own repository for a first run is a bad idea twice over: you do not know what it should do, so you cannot tell whether it did it. Use a fixture instead. GlassCoder ships eight, and the console host will write them to disk for you.

Do this

glasscoder fixtures --work .\.debug\fixtures

That materialises every suite task into its own folder and checks each one starts in the state its task claims. We will use suite-03-off-by-one, which is four files and one genuine bug: 

// .debug/fixtures/suite-03-off-by-one/Pager.cs
public int PageCount => ItemCount / PageSize;   // line 16

Integer division truncates, so 31 items at 10 per page reports 3 pages rather than 4, and 1 item reports 0 pages. Program.cs already asserts the correct answers, so the fixture starts red — and you will be able to tell, unambiguously, whether the agent fixed it.

Point the app at the fixture

The working directory is what decides which repository the agent reads and writes, so name it explicitly rather than relying on where you happen to launch from: 

# PowerShell
$env:GlassCoder__Workspace__RepoRoot = "$PWD\.debug\fixtures\suite-03-off-by-one"
dotnet run --project src/GlassCoder.Wpf
Do not press F5 yet

GlassCoder.Wpf has no launchSettings.json, so Visual Studio starts it in bin\Debug\net10.0-windows and the agent would explore its own build output. The reference guide has the profile to add if you want F5 to work.

01

First launch

The window opens on the Transcript surface with nothing in it. That is correct:

Read the status bar before anything else

It is the only honest inventory of what the model can call this session. Count the tools and notice what is missing — there is no bash, because Sandbox:EnableBashTool is false. If that line does not say what you expect, the app did not load the configuration you think it did, and every lesson below will behave strangely.

Sanity check

Seven tools, Cancel greyed out, empty grid. If instead you see a Failed: message, the app started but could not reach your endpoint — check Models:Roles:worker:Endpoint.

02

Your first goal

Start read-only. Workspace:WritablePaths ships empty, which means nothing is writable at all, so this run physically cannot change the fixture — a good property for a first look.

Type this, then press Run

Find the class that computes how many pages are needed, and tell me which file and line it is on.

The status bar switches to Running… and rows appear as each step completes:

What the last row tells you

Step 3 has - in Tools and Completed as its outcome. That is not a special "finished" record — it is how the loop ends. The model answered with text instead of calling a tool, so the loop treats the goal as met and stops. Every earlier step reads continued.

The Summary column shows the response text for that final step, and the tool name:status pairs for the others. succeeded is one of five registry statuses; the other four are failed, unknown_tool, invalid_arguments and faulted.

Status bar readsMeaning
CompletedThe model stopped calling tools. It believes it is done — which is not the same as being right.
StepLimitHit Agent:MaxSteps. Usually a loop that is going in circles.
ToolFailureLimitToo many consecutive unparseable calls. Look at Lesson 4.
TokenLimit · TimeLimit · CostLimitA budget in Agent: tripped.
CancelledYou pressed Cancel.
ModelErrorThe endpoint itself failed. The status bar shows Failed: ….

Tool-call validity 100 % is the number Phase 0 lives or dies by: every call the model made parsed and bound to a real tool. Below about 90 % nothing further is worth measuring, because the harness is spending its steps on malformed calls.

03

Reading one step

Do this

Click row 2 — the read_file step.

The pane underneath fills with the entire step: prompt, response, and tool results.

Four kinds of line appear, tagged by role: [system], [user] and [assistant] come from the prompt that was actually sent, and [tool name → status] blocks carry what each tool returned.

This is the prompt, not a summary of it

The recorded prompt is the exact window sent to the model for that step, after compaction and redaction. If a run went wrong because the model never saw something you assumed it saw, this pane is where you find out.

04

Catching a bad tool call

Sooner or later a model emits arguments that do not fit the schema. Those steps colour themselves amber, and you can isolate them without reading the run.

Do this

Set Severity to warning.

Where the colours come from

Nothing sets severity by hand — it is derived from the step record:

SeverityCondition
errorThe step carries an error — typically the model call itself failed.
warningAt least one tool call did not parse. This is the complement of tool-call validity.
infoEverything else.

So filtering on warning shows you precisely the calls that dragged the validity number down, and the detail pane shows the arguments the model actually sent. If you see a pattern — always the same missing field — that is a prompt or schema problem, not a model problem.

Clear is not a filter

The Clear button empties the transcript bus, not just the table. The run is still in the JSONL under Logging:Directory, but the live view is gone. Use the filters to narrow, and Clear only when you want a clean slate.

05

Letting it edit

Two settings have to change before an edit can land, and both default to refusing.

Do this, then restart the app

$env:GlassCoder__Workspace__WritablePaths__0 = "."
$env:GlassCoder__Approval__RequireApprovalForWrites = "true"

The first opts the fixture root into being writable — empty means nothing is. The second turns on the interactive gate; leave it false and writes land silently with no prompt at all.

Type this, then press Run

Pager.PageCount is wrong when the last page is partial. Fix it, then run the tests to prove the fix.

The run switches you to Transcript automatically. When the agent proposes its edit, the loop blocks and the Changes surface raises a banner. Click Changes:

Three things are true while that banner is up

  • The agent is genuinely blocked. The gate is awaited on the loop's thread; the banner is raised on the UI thread. Nothing proceeds until you click.
  • Silence is refusal. After Approval:ApprovalTimeoutSeconds — 300 by default — the change is rejected with Nobody answered within 300 seconds, so the change was not written.
  • The change already passed verification. The banner says so, and the summary under the grid shows it: Build of Pager.cs / No diagnostics. Approval is the last gate, not the only one.
Task id

Every desktop run is recorded under the task id desktop. That is how you will pick these runs out of metrics.jsonl later, and why suite runs get their own ids.

07

When verification refuses

Refused changes are recorded exactly like successful ones. A view that only showed writes that landed could not tell you what the agent tried to do — usually the more interesting question.

The Note column tells you who refused, and the wording is fixed:

NoteWho refused
Verification refused the edit.A rung of the ladder — the edit would have broken something that was not already broken.
A reviewer rejected this change.You, via the Reject button.
Nobody answered within 300 seconds…The approval timeout. An unattended prompt never becomes a write.
The run was cancelled while waiting for approval.You pressed Cancel with the banner up.
Why the summariser matters

One bad edit can produce hundreds of diagnostics that are all the same diagnostic. What reaches the model — and this pane — is the first error per file, deduplicated, capped and sorted earliest-first, with the true total always stated. The ratio between the true total and what was shown is the cascade ratio on the Metrics surface.

08

Comparing arms

One run tells you almost nothing. The question worth asking is which configuration is better, and by how much — so generate several arms with the console host, then read them in the app.

Do this

glasscoder ablate --work .\.debug\ablation

That runs the five standard arms: baseline plus one single-lever variant each. Then switch to Metrics in the app — selecting the surface reloads it from disk.

How to read this table

Each row differs from baseline by exactly one lever, so each column tells a story:

  • no-constrained-decoding — validity collapses from 97% to 71%, and pass@1 halves. The wasted steps show up as extra tokens. This is the arm that justifies the whole constrained-decoding apparatus.
  • no-prewrite-verification — the fastest arm, at 21.3s. It is also the one with nearly four times the compile errors per edit. Speed bought by writing broken code is not speed.
  • unsummarised-diagnostics — cascade ratio jumps to 6.8 and tokens more than double. The model is reading hundreds of consequences of one root cause.
  • no-context — validity is fine, pass@1 is not. The harness is working; the model simply does not know enough.
Two traps in this table

pass@1 shows -, not 0 %, when nothing in a group was graded against an oracle — ad-hoc desktop goals are never graded, only suite tasks are. And the bar is relative: it is scaled to the widest row in the current grouping, so it rescales when you regroup by Task or Source. Read the number, not the bar.

09

Exercises

Each of these teaches something the happy path cannot.

TryWhat you should observe
Press Reject on a good change Row turns red with A reviewer rejected this change. The loop resumes and the agent is told the write was refused — watch how it reacts.
Raise the banner and walk away Set ApprovalTimeoutSeconds to 20 first. The change rejects itself. Nothing is written.
Set Agent:MaxSteps to 3 The status bar reads StepLimit instead of Completed, mid-investigation. Budgets are hard stops, not suggestions.
Empty WritablePaths again, then ask for the fix Every edit is refused. Read-only is enforced by the workspace, not by asking the model nicely.
Run the same goal twice and group Metrics by Task Both runs land under desktop. This is why the suite exists — ad-hoc goals cannot be compared.
Press Cancel mid-run Cancelled. in the status bar, and any pending change rejected with the cancellation note rather than left undecided.
Point RepoRoot at suite-01-failing-test A different bug, same nine lessons. Eight fixtures ship, increasing in difficulty from one file to three.
