01

Which servers, and why

Three were considered. They are not interchangeable, and two of them answer genuinely different questions.

ServerAnswersVerdict
Microsoft Learn What is correct? Authoritative, versioned API surface. start here
GitHub What exists, and how is it used? Code and repository search across public GitHub — not just your own repository. close second
Stack Overflow Mostly what the other two answer better. skip

Why Learn first

It targets a failure mode this repository already documents and already tries to detect. A local worker with a training cutoff writes code against APIs it half-remembers; rung 2 catches the invented ones, but only after a step has been spent writing them. The Provenance subsystem exists to judge whether the context was newer than the code. Learn is the cheapest way to make the context newer, and its results carry a version — which matters more than it sounds, per the next paragraph.

Why GitHub is close, and where it is weaker

Public code search is a much larger claim on the “knowledge” slot than issues and pull requests. Learn tells you a signature; real code shows the idiom, the wiring, the three-type interaction that documentation describes badly. Learn also stops at Microsoft's surface — the moment the agent touches a third-party library, Learn has nothing and GitHub has everything.

Two properties make it weaker for this harness specifically:

  • Recency is not legible. A snippet does not say what it targets. Code written for an older framework looks identical to current code in isolation, and public repositories skew old because age is what accumulates stars. Retrieval intended to fix a stale cutoff can hand back equally stale code with nothing marking it as such.
  • There is no authority gradient. Learn has one answer. Code search has ten thousand, including abandoned, wrong, and increasingly model-generated code. Ranking by popularity is not ranking by correctness.
The narrow use that is unambiguously good

Code search is an excellent hallucination detector. Array.SortedCopy returns essentially nothing across public C#; Array.Sort returns millions. That is a cheap existence check on a symbol the model has just invented, it runs before a step is spent writing the file, and none of the ranking problems above apply to it — a count of zero is a count of zero. If GitHub earns its place, this is probably why.

Why Stack Overflow is out

Answers skew old, the accepted answer is often a decade past its usefulness, and the model has already absorbed most of public Stack Overflow during pretraining. You would be paying context-window rent to re-inject noisier copies of things it already knows. It is also worth confirming a suitable public MCP server exists at all before planning around one.

02

The hermeticity problem

This is the part that has to be designed first, because it is the part that quietly breaks the project rather than the feature. TaskSuiteDefinition states the property it depends on: 

// The fixtures live here as text rather than as files on disk so the suite is
// hermetic - every run starts from a byte-identical repository, which is what
// makes two ablation arms comparable at all.

A live network tool is byte-identical to nothing. Search results change between Tuesday and Thursday; a repository gets more stars; an article is revised. Two arms run a day apart stop being comparable, and the comparison is the reason this repository exists.

The cache is the feature

Record every MCP request and response to disk on first use; replay from the recording during suite and ablate. Three modes, one setting:

ModeBehaviourUsed by
LiveAlways call out; do not record.Interactive desktop work.
RecordCall out, write every exchange to the cache.Once, to build a corpus.
ReplayServe from the cache; a miss is an error, never a live call.Every suite and ablation run.

A miss in replay mode has to fail loudly. A cache that silently falls back to the network is a cache that gives you non-reproducible runs and the belief that they are reproducible, which is worse than not having one.

The payoff is larger than reproducibility. With a cache, “does retrieval actually help?” becomes a proper single-lever arm alongside no-context and no-constrained-decoding — the same question, asked the same way, answered with the same numbers. Without one, it is an anecdote.

Practical, not just philosophical

GitHub code search is authenticated and rate-limited. An ablation is dozens of runs over eight tasks. Replay is what stops the fourth arm from being throttled into a different shape than the first.

03

What it costs the tool contract

§7's invariant is that a tool's schema is generated from its C# signature, “so the model's contract cannot drift from the executor.” An MCP tool arrives with a server-declared schema and a server-side executor. The invariant weakens to “trust the server”, and there is no way to keep it fully. What can be kept:

  • Validate at registration. ToolFunctionFactory.ValidateSchema already rejects anything that is not a usable JSON object schema. Run the same check over advertised MCP tools and refuse to register the ones that fail, loudly, at startup — not on first use, mid-run.
  • Errors stay observations. A dead server, a timeout, a 429 must come back as ToolObservation.Fail with a stable code, never as an exception escaping into the loop. sandbox_unavailable has an obvious analogue here; a new upstream_unavailable is probably worth adding to ToolErrorCodes, since metrics group by those codes.
  • Namespace the names. Three servers will each want to be called search. Prefix on registration — learn_search, gh_search_code — and the transcript's tool filter keeps working.
  • Descriptions are prompt, not documentation. Server-authored tool descriptions go straight into the model's context and are written by someone optimising for a different agent. Budget for rewriting them locally.
04

The context budget

Context:MaxContextTokens is 48000, and deliberately below what any served model advertises — the config says why: “effective context lags advertised context, and dilution costs wall-clock on every step.” Every tool's name, description and parameter schema is in that window on every step of every run.

The harness has eight tools today. Two MCP servers could plausibly double or triple that, with descriptions written to sell the tool rather than to fit a budget. Two things follow:

  • Register a subset, not a catalogue. Take the two or three tools per server that do the job. There is no obligation to expose everything a server advertises, and a server's full surface is designed for a general agent, not a C# repair loop.
  • Watch tool-call validity, not just pass@1. Validity is the Phase 0 metric and it is the one most likely to degrade as the tool list grows — more tools means more ways to pick the wrong one. If it drops below about 90%, the retrieval is not paying for itself no matter how good the results are.
05

Where the network actually is

Worth stating plainly in the operator's guide, because it is easy to assume otherwise. Sandbox:AllowNetwork is false and Sandbox:Mode is Docker. Those govern repository code — what build, run_tests and bash may reach.

MCP clients would run in the harness process, not in the container. Their traffic does not pass through the sandbox seam and is not covered by that setting. The guarantee stays true — untrusted repository code still cannot phone home — but the harness itself becomes a networked process, and anyone reading AllowNetwork: false as “this run touched nothing external” would be wrong.

  • Secrets — a GitHub token — resolve through the existing ApiKeyEnvironmentVariable pattern rather than living in config files.
  • MCP responses land in transcripts, so they go through SecretRedactor like everything else. A token echoed in an error body is the obvious way to leak one.
  • Server processes started by the harness are a lifetime problem: they need to die with the run, including when it is cancelled.
06

Untrusted content

This is a real risk here rather than a theoretical one, and it got worse the moment create_file shipped. Retrieved text enters the context of an agent that can create and edit files in the workspace.

Microsoft Learn is a trusted publisher. Public GitHub categorically is not: a README, a code comment, a string literal in any repository on earth is attacker-controllable text that a search can surface. “Ignore your previous instructions” in a repository description is not a hypothetical attack, it is a cheap one.

  • Tag it as untrusted in the prompt. Retrieved content is quoted evidence, not instruction, and should be framed that way where it enters the window.
  • Never let retrieval alone justify a write. The verification ladder is unaffected by anything a search returned — a change still has to compile and pass. That is a genuine structural defence and it is already in place.
  • The approval gate is the backstop. Which is an argument for turning Approval:RequireApprovalForWrites on whenever retrieval is on.
  • Start read-only. Search and fetch, nothing that writes to an external system. No creating issues, no commenting on pull requests.
07

How to know it helped

The arms follow the existing pattern in StandardArms: one lever each, expressed as configuration overrides, read against baseline.

ArmLeverWatch
with-learnLearn tools registered, replay mode. pass@1 and the freshness metric. This is the arm the feature lives or dies by.
with-code-searchGitHub search tools registered, replay mode. pass@1 and compile errors per edit — the hallucination-detector hypothesis.
retrieval-no-cacheLive mode instead of replay. Run-to-run variance. If it is large, every uncached result you have ever read was an anecdote.
tools-onlyMCP tools registered but stubbed to return nothing. Tool-call validity and tokens. Isolates the cost of having the tools from the value of their answers — the thing a naive comparison confounds.

That last arm is the one most people would skip and it is probably the most informative. If tools-only is meaningfully worse than baseline, retrieval has to clear that deficit before it breaks even, and you would want to know the size of the hole before reading any result as a win.

The suite may need a task that requires retrieval

All eight fixtures are self-contained by design and none needs external knowledge, so a perfect retrieval tool would score exactly zero improvement on the current suite. Proving the feature helps needs a task whose answer is not in the repository and not reliably in a local model's weights — a current API used correctly. That fixture does not exist yet, and writing it is part of this work rather than an afterthought.

Acceptance

  • A run in replay mode with a cold cache fails loudly and never reaches the network.
  • Two ablation runs of the same arm, days apart, produce identical retrieved content.
  • An unreachable MCP server produces a failed observation and a continuing run, never an exception out of the loop.
  • Registering the Learn tools does not drop tool-call validity below the Phase 0 gate on the existing suite.
  • No retrieved content ever appears unredacted in a transcript when it contains something the redactor recognises.
