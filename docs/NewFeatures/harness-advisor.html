The harness advisor

A headless Claude reading finished transcripts and metrics, not to judge a change, but to judge GlassCoder — to find the things the agent could not do, the tools that misled it, and the features that are missing. Its output is proposed work, not advice.

Nothing here is implemented. Where this page quotes the repository, that is current code. Proposed configuration keys, verbs and schema fields are marked as proposed.

00

The gap this closes

Until recently GlassCoder had no way to create a file. edit_file could change what existed and nothing could add anything, so the suite's own suite-07-feature-three-files — which asks for “a Discount type with a percentage” and ships no Discount.cs — could only be passed by cramming a new type into an existing file. That task was measuring the tool set rather than the model, and had been since edit_file landed.

It was found by a person, by accident, while writing a tutorial.

But it was never hidden. Every run that reached for a file that was not there would have written this into the transcript: 

[tool edit_file → Failed] 'Discount.cs' does not exist.
  code: not_found

One line, in a schema built to be machine-read, repeating across runs and across tasks. Nothing was reading for it. That is the gap this proposal closes: the harness records everything it would need to notice its own deficits, and nothing currently looks.

The distinction that matters

This is not a tuning tool. A misconfigured lever is what an ablation arm tests, and the critic proposal covers judging individual changes. This one is aimed at the class of problem no configuration change can fix — the agent tried to do something the harness cannot do at all — and its intended output is a new tool, a new rung, a new field, or a fixed contract.

01

What it looks for

Five classes of finding, ordered by how much they are worth. The first is the reason to build this; the rest come almost free once something is reading the logs at all.

ClassLog signatureOutput
Capability deficit
highest value		 The same tool failing the same way across unrelated tasks, or a visible workaround — a sequence of calls that achieves clumsily what one missing tool would do directly. A proposed tool or subsystem, as a workplan task.
Contract defect InvalidArguments clustered on one parameter of one tool; or a tool chosen for a job it does not do, which is a description problem wearing a model problem's clothes. A schema or [Description] fix — the cheapest change in the harness.
Missing oracle Runs that pass every rung and then fail the task oracle. Something was wrong that nothing in the ladder can see. A proposed rung, or a fixture that pins the behaviour.
Observability blind spot The advisor forms a hypothesis it cannot test, because the field it needs is not in the schema. A proposed StepRecord or RunMetrics field.
Harness lever
lowest value		 A configured value that looks wrong for the observed behaviour. A proposed ablation arm, tested by machinery that already exists.

The workaround signature is the interesting one

A capability deficit does not always announce itself as an error. Sometimes the agent finds a way around, and the way around is the evidence. The missing-file case had a textbook example: with no create tool, the only route to a new class was to append it to a file that already existed. That produces an edit_file whose oldText is the end of one type and whose newText declares a second — a shape that is detectable, unusual, and means “I wanted a new file.”

Errors are easy to grep for. Workarounds need something that understands what the agent was trying to do, which is precisely the thing a model is better at than a query.

02

Two it should have caught

Both of these were real, both were live for months, both were found by hand, and both left a machine-readable trail. They are the acceptance test for this feature: an advisor that cannot find these on historical logs is not worth running.

Case 1 · No tool could create a file

Signatureedit_file → not_found, message ending does not exist., on paths that never appear in any successful read_file.
CorroborationIt recurs across tasks with nothing else in common, which rules out one broken fixture.
The tellThe tool's own hint said “or ask for a file-creation tool” — the executor was documenting its own deficit in every failure message it emitted.
Correct outputPropose create_file. Note that suite-07 cannot be passed cleanly without it, so the suite is measuring the tool set.

Case 2 · The pre-write check refused correct new code

Rung 2 compiles in memory without running MSBuild, so it never saw the global usings the SDK generates for ImplicitUsings — that file lives under obj/, which the workspace deny list excludes. Any new code touching System was refused before it reached disk.

Signatureverification_failed whose diagnostics are overwhelmingly CS0103 on well-known framework names — Array, Console, ArgumentNullException.
CorroborationA build in the same run succeeding on code rung 2 called broken. Two oracles disagreeing is a strong signal, and both verdicts are in the same transcript.
Why a human missed itEach refusal looks locally reasonable. The pattern only exists across runs.
Correct outputPropose synthesising the SDK's global usings when the project asks for implicit usings — or, failing that diagnosis, at minimum flag that rung 2 and build disagree systematically.
Two oracles disagreeing is the highest-value pattern in the logs

It cannot be a model failure. When a cheap rung refuses something an expensive rung then accepts, one of them is wrong about the world, and the transcript holds both verdicts side by side. If the advisor only ever looked for this one pattern it would still earn its place.

03

What it reads

Nothing new. TranscriptReader states the property this whole proposal rests on: 

// if the log cannot be read back into a complete run, then the log was never a
// transcript. Everything downstream - the live transcript view, the metrics
// harness, the ablation runner - reads through here.

The advisor is one more thing reading through there, over four artifacts:

ArtifactCarries
StepRecordThe prompt as actually sent, the response, every ToolCallRecord with arguments, status and result, ContextCompacted, EstimatedContextTokens, Outcome, Error, and the plan as it stood.
RunRecordGoal, stop reason, totals, config hash, commit.
RunMetricsTwenty-odd indicators including validity rate, compile errors per edit, recovery rate, cascade ratio and Arm.
ChangeLogWhat was proposed versus applied versus refused, and by whom.

Offline, and that is a design decision

  • It cannot contaminate a measurement. An observer able to influence a run changes what the run measures. One reading finished JSONL cannot.
  • It sees many runs at once. Every case above is a cross-run pattern. A per-run observer would have missed both, for the same reason the humans did.
  • It is re-runnable. Improve the analysis and re-run it over months of history at no cost to any live run.

Why headless Claude Code rather than an API call

Volume. Prompts are logged up to MaxLoggedTextLength — 16 000 characters — and an ablation is dozens of runs. That does not fit in a request.

What fits is an agent that starts from the skeleton — steps, tool names, statuses, outcomes, metrics — spots something odd, and then pulls the full prompt for those specific steps. A read-only loop over a log directory with grep and read is exactly the shape claude -p is good at, and exactly the shape a single Messages call is bad at. This is the inverse of the recommendation in the critic proposal, and for the opposite reason: there, latency was bounded and context was small.

04

Proposals, not advice

The failure mode to design against is obvious once stated. A model handed transcripts will produce ten confident, plausible, mostly generic suggestions — improve your tool descriptions, consider more context — and evaluating them will cost more than ignoring them would have. That is also flatly at odds with how the rest of this repository works, where every claim gets a lever and a number.

So constrain the output shape. A finding is not prose; it is a candidate workplan task with evidence attached, in the format WORKPLAN.md already uses. 

{
  "class":     "capability-deficit",
  "title":     "Implement create_file with guardrail and pre-write verification",
  "estimate":  "1d",
  "dependsOn": [8, 14, 27, 28],

  "evidence": [
    { "runId": "…", "task": "suite-07", "step": 6,
      "quote": "'Discount.cs' does not exist.", "code": "not_found" },
    { "runId": "…", "task": "desktop",   "step": 4,
      "quote": "'Helpers.cs' does not exist.",  "code": "not_found" }
  ],
  "occurrences": 23,
  "tasksAffected": ["suite-07", "desktop"],

  "rationale":  "edit_file cannot create files. Its own hint names the missing tool.",
  "acceptance": "A new file is created through the change log and the approval gate;
                 an existing path is refused rather than overwritten.",
  "predicted":  { "metric": "pass@1", "task": "suite-07", "direction": "up" }
}

Four properties of that shape are doing real work:

  • Evidence is quoted, with run and step ids. Every claim is checkable in ten seconds against the transcript view. A finding whose evidence does not resolve is a hallucination, and it is trivially detectable as one.
  • Occurrences are counted. One instance is an anecdote. Twenty-three across two unrelated tasks is a deficit. The number is what turns a hunch into a priority.
  • There is a prediction. Which makes the proposal falsifiable, and lets section 5 work at all.
  • It lands in the existing plan. A candidate task with an estimate, dependencies and acceptance criteria is a thing you can accept or reject in one reading. Prose is not.
05

Scoring the advisor

The advisor is a measurement instrument pointed at a project whose whole discipline is measuring before believing. It would be absurd to exempt it. Because every finding carries a prediction, it can be scored on its own terms:

MetricDefinitionReading
Adoption rateFindings accepted into the workplan ÷ findings raised.Low means it is generating noise you have to wade through.
Hit rateImplemented findings whose predicted metric moved in the predicted direction ÷ implemented findings. The real number. Below roughly a third, it is guessing.
Evidence integrityFindings whose quoted run and step ids resolve to the quoted text. Must be 100%. Anything less is fabrication, and grounds for distrusting the rest of the report.
Lead timeHow long the earliest evidence for a finding sat in the logs before the advisor raised it. Run against history, this is the feature's whole argument, in months.

Evidence integrity is the one to enforce mechanically rather than measure: the report writer should resolve every citation against the transcript before emitting it, and drop findings that fail. A fabricated quote is not a low-quality finding, it is a non-finding — and one is enough to make a reader stop trusting the other nine.

06

What it cannot see

Four honest limits, and the last is the sharpest.

  • Its ceiling is the schema. If something matters and is not in StepRecord, no analysis surfaces it. This is why “propose a new field” is a first-class output class rather than an afterthought.
  • It only sees what the agent bumped into. A deficit becomes visible when the agent tries and fails. Nothing in the logs reveals a tool nobody has yet needed, so the advisor is a good detector of friction and a poor source of vision.
  • Some defects never reach the logs at all. Two real examples from this repository: the critic panel counting an unreachable critic as non-refuting, and the analyzer throwing when walking a directory that does not exist yet. The first is behind a disabled feature; the second was unreachable until create_file existed. Neither would appear in any transcript. The advisor complements code review; it does not replace it.
  • There is a conflict of interest. A large cloud model advising on a harness whose thesis is that a local model can do the work may drift toward proposals that suit a large cloud model — more tools, more context, more capability. The prediction and the hit rate are the defence: a proposal that does not move the number is rejected however sensible it sounded.
Transcripts contain your source code

Logging:LogSourceContent is true by default and prompts carry repository content, so pointing a cloud model at a log directory sends that code outward. The config already documents the switch — false keeps “the step skeleton” and drops all source content — so a redacted advisor mode is half-built. That mode still finds both cases in section 2, because both are visible in tool names, error codes and diagnostic ids rather than in source.

07

Where it must not go

“GlassCoder improves itself” has an obvious next step, and it is the wrong one.

TemptingWhy not
Let the advisor's findings be implemented by GlassCoder, on GlassCoder A measurement instrument that rewrites itself in response to its own measurements stops being an instrument. When pass@1 moves you would no longer know whether the harness improved or the yardstick bent. Findings are proposals for a human to accept — that is the entire boundary, and it is worth being rigid about.
Run the advisor live, so it can interrupt a bad run That is a supervisor, not an advisor, and it is a different feature — Orchestrator.cs and Orchestration:Enabled already sit at false for Phase 5. Conflating them gives you an advisor that influences the runs it reports on.
Give the advisor write access so it can open issues or branches Read-only is what makes it safe to point at every log you have. The report is an artifact; what happens to it is a separate, human decision.
Let it grade individual runs The suite oracle already does that, deterministically and for free. The advisor's subject is the harness, and every step toward judging runs is a step away from the thing nothing else is doing.

The one-line version, worth putting in the class comment: the advisor proposes, humans dispose, and it never touches the repository it is reporting on.

Shape of the work

Acceptance

  • Run against logs predating create_file, the advisor raises a capability-deficit finding for a file-creation tool, with citations that resolve.
  • Run against logs predating the implicit-usings fix, it raises the rung 2 / build disagreement.
  • Every citation in every emitted finding resolves to the quoted text. No exceptions and no partial credit.
  • The advisor never opens a file in the repository under analysis for writing, and its report is the only artifact it produces.
  • In redacted mode, no repository source content appears anywhere in the report.
