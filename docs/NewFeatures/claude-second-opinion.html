This describes something that does not exist

Nothing on this page is implemented. It is a design note for a future version, written against the code as it stands so the estimates and the names are real. Where it quotes the repository, that is current; where it quotes configuration keys that would be added, they are marked as proposed.

00

What already exists

More than you would expect. CriticPanel is rung 6 of the verification ladder and it is fully written: it fans out N critics in parallel, gives each a different lens, parses a JSON verdict, and counts votes against a threshold.

PieceWhereState
Multi-critic refutationCore/Verification/CriticPanel.cs Written. Critique:Enabled is false.
Ladder integrationCore/Verification/VerificationLadder.cs Written. Rung 6, runs last, skipped when critique is off.
An arm that turns it onLab/Ablation/StandardArms.cs Written — with-critique. Not in Default.
Hosted-role configurationModels/Configuration/ModelRoleOptions.cs Written: ApiKey, ApiKeyEnvironmentVariable, per-role cost.
A non-OpenAI transportModels/ChatClientFactory.cs Missing. This is the whole gap.

The critic prompt already asks for the right thing. Each critic is told to refute rather than review, which the class documents as deliberate: 

// CriticPanel.cs
// "is this good?" invites agreement from a model that has just been shown a
// plausible-looking diff, while "find what is wrong with this" gives it a job
// it can fail at honestly.

And the three lenses are already split so the critics are not redundant: correctness (does it do what the goal asked, in every case the goal implies), regression (does it break behaviour other code depends on) and evidence (does the evidence prove the change works, or only that it compiles).

The argument is already in your codebase

CritiqueOptions.Role is documented as “ideally a different model family from the worker, so the critic's blind spots are not the worker's blind spots.” That is the case for this feature, written before the feature was proposed. A local worker critiqued by a second instance of itself mostly agrees with itself.

01

Two features, not one

“Let Claude check a proposal” collapses two different things. They differ in who the advice is for, and that changes almost every design decision downstream.

A · Critic roleB · Ask Claude
AdvisesThe agentYou
WhenEvery change, automaticallyOnly when you press it
WhereRung 6 of the ladderThe approval banner
Consumed byThe next model stepA human reading a diff
Latency budgetTight — it is in the loopLoose — someone is already waiting
OutputStructured verdict, machine-readProse, human-read
Cost profileEvery change × N criticsOne call, on demand
EffortSmall — a transportMedium — new UI and gate semantics

They are complementary rather than alternatives. A is a better oracle; B is a better review experience. Build A first because it is nearly free, but B is the one that changes how the app feels to use.

02

A · Claude as the critic role

In the ideal case this is a configuration change and nothing else — which is exactly what ModelRoleOptions promises: “swapping a local server for a hosted API … must never be a code change.” 

// config/appsettings.json — the shape it should end up as
"critic": {
  "Provider": "anthropic",              // proposed: new field
  "Endpoint": "https://api.anthropic.com",
  "ModelAlias": "claude-opus-4-8",
  "ApiKeyEnvironmentVariable": "ANTHROPIC_API_KEY",
  "Temperature": 0.0,
  "InputCostPerMillionTokens": 0.0,     // fill in from current pricing
  "OutputCostPerMillionTokens": 0.0
}

The gap is that ChatClientFactory builds one shape of client: Microsoft.Extensions.AI.OpenAI over an OpenAI-compatible base endpoint. The pipeline it assembles — role defaults, then constrained decoding, then OpenTelemetry, then transport — is worth keeping intact, so the change is a transport swap at the innermost layer, not a parallel factory.

  • Add Provider to ModelRoleOptions, defaulting to openai so every existing config keeps working untouched.
  • Branch only on the innermost client. Everything outside it — telemetry, role defaults, cost accounting, the deliberate absence of UseFunctionInvocation() — stays shared.
  • Constrained decoding does not apply. The critic returns JSON by prompt, not by grammar, and CriticPanel.Parse already falls back to a text heuristic when the JSON is malformed. Leave the stage in the pipeline; it will simply attach nothing for this provider.
Cost is already instrumented

Per-role InputCostPerMillionTokens and OutputCostPerMillionTokens feed the cost-per-solved-task metric, and Agent:MaxCostUsd is an existing hard budget. A paid critic slots into both without new plumbing — which also means a runaway critique loop trips a budget rather than a credit card.

03

B · Ask Claude at the gate

A third button on the approval banner, next to Approve and Reject. It does not decide anything — it fills a pane underneath with an opinion, and leaves the decision where it was.

Three things the implementation has to get right

  • The button must not complete the TaskCompletionSource. The loop thread is parked inside WpfApprovalGate.RequestAsync awaiting ChangesViewModel.RequestApprovalAsync. Only ApproveCommand and RejectCommand may call TrySetResult. A third command runs a side query and raises a property.
  • The approval timeout is a trap. WpfApprovalGate starts a CancelAfter(ApprovalTimeoutSeconds) — 300 by default — the moment the banner goes up, and expiry is a rejection. A Claude round-trip plus your reading time can cross that and silently discard the change. The timer needs to be suspended while an advisory query is in flight, or reset when one returns. This is the single most likely thing to be got wrong.
  • Cancellation still has to win. Pressing Cancel during an advisory query must abandon it and reject the change with the existing wording, The run was cancelled while waiting for approval.

What the query gets is the interesting design question. The minimum is what rung 6 already assembles: the goal, the diff, and the verification evidence. That is enough to catch the example above. It is not enough to catch “this duplicates a helper three files away” — which is the argument for the second option in the next section.

04

API or Claude Code headless

“Headless Claude” could mean two quite different things, and the choice does not have to be the same for A and B.

Messages APIClaude Code headless
SeesOnly what you put in the promptThe repository, by reading it
IntegrationAn IChatClient behind the existing seamA process invocation
LatencyOne round tripIts own agent loop
CostPredictable per callUnbounded until it stops
CatchesErrors visible in the diffErrors visible only in context
Sandbox storyNetwork call from the harnessA second agent with file access

Recommendation: Messages API for A, because rung 6 runs on every change and needs a bounded cost and latency. Consider the CLI only for B, where a human is already waiting and depth is worth seconds — and where a single unbounded call is a decision someone just made deliberately by pressing a button.

The CLI option is a second agent, not a second opinion

Claude Code headless has its own tools, its own loop and its own budget. Nothing in GlassCoder's transcript, metrics or change log would see inside it, so a run that used it would have a hole in exactly the place this project exists to make visible. If it is used at all, it should be read-only and its output treated as text — never as a tool call, and never as something that can write.

05

What must not change

This is the part worth being stubborn about. The project's claim is that a local model, driven by a good harness, can do useful work. A remote critic is compatible with that claim, but only under conditions.

  • The worker stays local. The thing being measured is the worker. If Claude ever writes code that lands, the measurement is gone and the project is answering a different question.
  • Critique:Gates stays false. Your own comment gives one reason — “a model refuting a compiler-verified change is a claim worth reading rather than obeying.” There is now a second: the moment a remote critic can veto, “the local model solved suite-03” stops being a clean result.
  • Suite and ablation runs default to critique off. A hosted critic makes runs non-reproducible and costs money per arm. with-critique stays a named arm you opt into, not part of StandardArms.Default.
  • The critic never sees secrets. Whatever is sent goes through the existing SecretRedactor path, not around it.

There is a positive version of this too. Because critique does not gate, turning it on cannot change pass@1 — so the honest way to justify the feature is to show it moves recovery rate: how often the agent gets out of a hole it has dug. That number already exists on the Metrics surface, and with-critique already exists as the arm that measures it.

06

Fix this first

CriticPanel.AskAsync catches a failed critic and returns a verdict. The comment above it says what it intends: 

// A critic that cannot be reached must not silently become an approving vote.
_logger.LogWarning(ex, "Critic {Index} failed; counting it as a refusal to judge");
return new CritiqueVerdict(false, 0d, $"Critic unavailable: {ex.Message}");

The verdict is recorded with Refuted = false. The tally counts refutations and compares them to a threshold, so a critic that could not be reached is arithmetically indistinguishable from one that read the change and accepted it — and the summary string reports it under “N/M critics accepted the change”. With three critics and a threshold of two, two unreachable critics and one genuine refutation produce “not refuted”.

Today this is nearly unreachable: the critic is a local endpoint that is either up or the whole run fails. Point it at a hosted API and unreachable critics become routine — rate limits, timeouts, a tripped cost budget, an expired key. So this wants fixing before the transport lands, not after.

OptionBehaviour
QuorumRequire a minimum number of real verdicts; below it, report the panel as inconclusive rather than as accepting. Matches the comment's intent most closely.
Fail closedAn unreachable critic counts as refuting. Safe, but a flaky network becomes a stream of false refutations — and with Gates = false those are noise in the agent's context.
Report onlyKeep the tally, but never let a panel with unreachable critics claim consensus in its summary text.

Quorum is the one I would pick, with the count of unavailable critics surfaced in CritiqueResult so the transcript shows how much of the panel actually voted.

07

Decisions needed

QuestionWhy it matters
Does B share the critic role, or get its own? Sharing means one config and one cost line. A separate role lets the interactive one be a larger, slower model, since a human is waiting rather than a loop.
Does B's opinion reach the agent? If you reject after reading it, the agent is told the write was refused but not why. Feeding the text back as the rejection reason would make it act on your reviewer's reasoning — powerful, and a change to what rejection means.
Is B's output recorded? Everything else on the Changes surface is. An advisory opinion that shaped a decision but left no trace is the one thing on that surface you could not reconstruct afterwards. It probably belongs on CodeChange.
How much context does B send? Diff only is cheap and catches diff-visible errors. Diff plus the files it touches catches more and costs more. There is no reason both cannot be offered.
What happens with no API key? The button should be absent or disabled, not fail on press. Enabled on ICriticPanel already models exactly this for rung 6.

Acceptance, in the style of the existing plan

  • A panel with fewer real verdicts than its quorum reports inconclusive, and never reports consensus.
  • Switching the critic role from a local endpoint to a hosted one is a configuration change with no code change and no test change.
  • The Ask Claude button never resolves the approval gate, and a change left with an opinion on screen for longer than ApprovalTimeoutSeconds is still awaiting a decision rather than silently rejected.
  • pass@1 is unchanged by enabling critique; if it moves, something gates that should not.
