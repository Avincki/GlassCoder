00

Two verbs that write

Start from the status bar. It is the only honest inventory of what the model can call:

Ready. 8 tools: update_todos, read_file, grep, glob, create_file, edit_file, build, run_tests

Two of those eight change files, and they are deliberately not one tool:

  • create_file(path, content) writes a whole new file, and refuses to overwrite. Anything already at that path is an error.
  • edit_file(path, oldText, newText) replaces one exact, unique string in a file that already exists, and errors if the target is missing or appears more than once.

Keeping them apart is the point. edit_file's guarantee is that an edit can never land silently in the wrong place — and an upserting create tool, one that quietly replaced a file's whole contents when it happened to exist, would be a hole straight through that guarantee. So the failure modes are sharp, and you will meet all four:

You ask forCodeWhat comes back
create_file on a path that exists already_exists 'DoubleSorter.cs' already exists.
edit_file on a file that does not not_found 'DoubleSorter.cs' does not exist.
edit_file whose target text is absent not_found The text to replace was not found in 'DoubleSorter.cs'.
edit_file whose target occurs twice ambiguous_target The text to replace appears 2 times in 'DoubleSorter.cs'.

The already_exists hint points straight at the other verb — “Use edit_file to change it. This tool will not overwrite a file.” — which is what the agent needs to recover, and what it will do on its second attempt after a rejected approval.

Why not just give it a shell?

bash could write a file in one line, but it is not registered at all unless Sandbox:EnableBashTool is set, it then refuses to run unless the container sandbox is available, and — the part that matters here — it does not go through the change log. A file created with bash produces no proposal, no diff, no approval prompt and no row on the Changes surface.

create_file takes the same five steps as an edit, in the same order: the path allow-list, the change log, the pre-write compile check, the approval gate, then the write. That is the whole reason it exists as a first-class tool rather than a shell command. A surface that cannot see a write is a surface that lies.

01

Write the spec first

“Write a class that sorts an array of doubles from low to high” sounds complete and is not. It leaves at least four decisions open, and the agent will make all four of them without telling you. So write them down as code that runs.

GlassCoder's own task suite has a convention for exactly this, and it is worth copying: a self-contained console project with no package references, whose entry point asserts the required behaviour and returns a non-zero exit code when it does not hold. Exit code zero and the words ALL TESTS PASSED are the oracle. No test framework to restore, nothing to interpret.

Do this

glasscoder fixtures --work .\.debug\fixtures

Then borrow two files from any suite task, so the harness and the project file are the shipped ones rather than ones you retyped. Three files in total: the agent writes the fourth.

mkdir .debug\sorter
copy .debug\fixtures\suite-03-off-by-one\Check.cs .debug\sorter\
copy .debug\fixtures\suite-03-off-by-one\Fixture.csproj .debug\sorter\

Check.cs is thirty lines: Check.That, Check.Equal and Check.Exit. Now write the spec. 

// .debug/sorter/Program.cs
using Fixture;

double[] input = [3.5, -1.25, 10, 0, 2];
double[] sorted = DoubleSorter.Ascending(input);

Check.Equal(5, sorted.Length, "every value is returned");
Check.Equal(-1.25d, sorted[0], "the smallest value comes first");
Check.Equal(10d, sorted[4], "the largest value comes last");
Check.Equal(3.5d, input[0], "the input array is not reordered");
Check.Equal(0, DoubleSorter.Ascending([]).Length, "an empty array sorts to an empty array");

double[] withNaN = DoubleSorter.Ascending([1, double.NaN, -1]);
Check.That(double.IsNaN(withNaN[0]), "NaN sorts before every number");

try
{
    DoubleSorter.Ascending(null!);
    Check.That(false, "a null input throws ArgumentNullException");
}
catch (ArgumentNullException)
{
    Check.That(true, "a null input throws ArgumentNullException");
}

return Check.Exit();

Seven checks, and four of them exist only because the English sentence was ambiguous:

The sentence does not saySo the spec says it
Whether the input may be reordered in place It may not — input[0] is still 3.5 afterwards.
Where NaN goes First. That is what Array.Sort does; pinning it makes it a decision rather than an accident.
What an empty array does Returns an empty array, rather than throwing.
What null does Throws ArgumentNullException.

This file also fixes the API: the class is DoubleSorter, the method is Ascending, it takes a double[] and returns one. The agent is writing an implementation, not designing a surface — and the difference between those two jobs is the difference between a run you can grade and a run you cannot.

Confirm it starts red

The spec calls a class that does not exist, so nothing compiles. That is the point — a fixture that already passes measures nothing: 

cd .debug\sorter
dotnet build

Program.cs(4,19): error CS0103: The name 'DoubleSorter' does not exist in the current context
Program.cs(10,16): error CS0103: The name 'DoubleSorter' does not exist in the current context
Program.cs(12,20): error CS0103: The name 'DoubleSorter' does not exist in the current context
Program.cs(17,5): error CS0103: The name 'DoubleSorter' does not exist in the current context
A broken repository does not block the agent

Four errors before the agent has done anything might look like trouble for a harness that refuses writes which break the build. It is not. The pre-write check compiles the project with and without the new file and only refuses errors the file itself introduces. The most common reason to create a file is that something already refers to it, so a rule that counted these four would refuse every useful creation.

02

Pointing the app at it

Three settings, all of which default to something safer than what this run needs. Set them in the shell you launch from — the working directory is what decides which repository the agent reads and writes, so name it rather than relying on where you happen to be: 

# PowerShell
$env:GlassCoder__Workspace__RepoRoot = "$PWD\.debug\sorter"
$env:GlassCoder__Workspace__WritablePaths__0 = "."
$env:GlassCoder__Approval__RequireApprovalForWrites = "true"

dotnet run --project src/GlassCoder.Wpf
SettingShips asWhy it has to change
Workspace:RepoRoot. Otherwise the agent works on whatever directory you launched from.
Workspace:WritablePaths[] Empty means nothing is writable, and both write verbs come back path_not_allowed.
Approval:RequireApprovalForWritesfalse Left false, the write lands silently. You want to see the diff this time.

All three are also on the Settings… tabs, and saving them there makes them stick across sessions rather than dying with the shell — the settings guide has the details. Environment variables are used here precisely because they do die with the shell, and because they outrank anything the dialog saved, which keeps this lesson reproducible on a machine that has been configured for something else.

03

The goal and the plan

Type this, then press Run

Program.cs describes a class called DoubleSorter that does not exist yet. Read the checks, create DoubleSorter.cs with the class in it, then build to prove it compiles.

Notice what the goal does not contain: the algorithm, the signature, or the edge cases. They are all in Program.cs, and telling the agent to go and read them is cheaper — and far less likely to drift from what the oracle actually checks — than restating them in the prompt.

Two things worth knowing about that first row

  • The plan is stored on every step, not once. Each step record carries the plan as it stood at that moment, so a plan the agent quietly abandoned halfway is visible afterwards — you can see the item that went from InProgress back to nothing.
  • update_todos replaces the whole list. There is no per-item mutation, and it refuses a list with two items marked InProgress: 2 items are InProgress. Work on one thing at a time.
04

What the check still refuses

Before the file exists at all, its content is compiled in memory. Two kinds of problem stop it, and the first line of the message tells you which:

Array.SortedCopy is a plausible-looking method that .NET does not have. A syntax check cannot see that; only a compile can. Catching it here rather than at build saves the loop a step, and — more usefully — the file never reaches disk, so there is no half-written class for a later step to trip over.

First line readsWhich means
Syntax check of the new file failed. The content does not parse. Reported against the repo-relative path, as DoubleSorter.cs(5,58): error CS1525: Invalid expression term ';'
This file would introduce N new compile error(s). It parses but does not compile. Reported against the full path, because the compile sub-check is handed the file's absolute location.

Three details in that one row

  • Nothing was written — not even an empty file. The change is on the Changes surface as Rejected, noted Verification refused the edit., with the diagnostics attached. There is no DoubleSorter.cs on disk to clean up.
  • It is Failed, not a warning. The call parsed, bound and ran — the tool simply refused the job. So the row is info severity and does not dent tool-call validity. Filtering Severity by warning, which is how tutorial one found bad calls, will not find this. Filter by Tool instead.
  • More errors than are listed. The summariser keeps the first error per file, so a second error in the same file is counted in the total and not shown. That gap is the cascade ratio on the Metrics surface.
Rung 2 is approximate on purpose

It parses the C# under the project directory and compiles it against the harness's own reference assemblies rather than resolving a real MSBuild graph, which would cost seconds on every write. It catches hallucinated APIs, wrong signatures and type errors in well under a second; it is not a substitute for build, which remains the authoritative compile gate and is what the next lesson runs.

05

A diff that is all green

When the content does compile, the loop blocks on the approval gate instead. Click Changes:

Reading a creation as a diff

  • Every line is green and there is no context. A creation is diffed against an empty before, so there is nothing unchanged to show either side of it. If you see even one red line here, the tool was edit_file and something already existed.
  • The Lines column reads 1-18, computed from the diff rather than reported by the tool: the lowest and highest line touched by anything that is not context. For a creation that is always the whole file.
  • The line under “Diff — DoubleSorter.cs” is blank. That is the verification summary, and on a clean edit the pre-write check returns nothing to say. Blank means checked and silent; text there means something was refused or the check could not run. An empty line is good news, which is worth internalising because it does not look like good news.
Do this

Press Approve. The row turns green and reads Applied, and the loop continues — it has a build to run.

The observation the model gets back is one line: Created DoubleSorter.cs (18 lines). — plus a payload carrying the path, the line count and "verified": true, meaning the pre-write compile check actually ran rather than being skipped over as it is for a non-C# file.

No using System;, and that is worth noticing

The class calls ArgumentNullException.ThrowIfNull and Array.Sort with no using directive in sight, because the project has ImplicitUsings enabled. The in-memory check knows that: it reads the project file and synthesises the SDK's global using set, since the generated one lives under obj/, which the workspace deny list excludes. Without that, every new file touching System would be refused — which is exactly the shape of file this tool exists to write.

06

Proving it three ways

The agent's own verdict is the weakest evidence available. There are three stronger ones, and they are not equally strong either.

1 · build — real, and authoritative

build shells out to dotnet build inside the sandbox and reports Build succeeded (0 warnings). The four CS0103 errors from lesson 1 are gone, which is genuine evidence: the class exists, it compiles against the real MSBuild graph rather than rung 2's approximation of it, and Program.cs can see it.

If Docker is not running

build and run_tests execute repository code, so they go through the sandbox seam — Docker by default, with no silent fallback to the host. You get build:Failed carrying Docker is not reachable, and running repository code on the host is not permitted. Start Docker, or set GlassCoder:Sandbox:AllowUnsandboxedExecution to true to accept the risk. The in-memory pre-write check needs none of this, which is why the file still got written.

2 · run_tests — a row that proves nothing here

run_tests runs dotnet test. This fixture is a console application with no test framework in it, so there is nothing for dotnet test to find, and the row still reads run_tests:Succeeded.

That is not a bug, and it is the single most misread thing on the transcript. Succeeded describes the call, not the code. It means the arguments bound, the tool ran, and it handed back an observation. The result has two levels, and they disagree on purpose: 

{"ok":true,"tool":"run_tests","summary":"0 of 0 tests failed.",
 "data":{"path":".","ok":false,"passed":0,"failed":0,"skipped":0,"total":0, … }}
   ↑ the call worked          ↑ the run did not

Zero of zero, out of a project with no tests in it. The rule generalises: a failing build also arrives as build:Succeeded, with Build failed with 3 error(s). as its summary and "succeeded": false in the payload — because a failed build is information the agent should act on rather than a tool malfunction. The five statuses — Succeeded, Failed, UnknownTool, InvalidArguments, Faulted — describe the call. Read the payload for the outcome.

3 · The oracle — the only one that grades behaviour

Neither a clean build nor a green tool status says the sort is correct. Run the fixture yourself, outside the app, exactly as the suite runner does: 

cd .debug\sorter
dotnet run

  pass  every value is returned (expected 5, got 5)
  pass  the smallest value comes first (expected -1.25, got -1.25)
  pass  the largest value comes last (expected 10, got 10)
  pass  the input array is not reordered (expected 3.5, got 3.5)
  pass  an empty array sorts to an empty array (expected 0, got 0)
  pass  NaN sorts before every number
  pass  a null input throws ArgumentNullException
ALL TESTS PASSED

Exit code 0, and the words ALL TESTS PASSED in the output. Those two together are what glasscoder suite requires before it records a pass, and they are the reason the fixture convention is worth following for one-off work like this: you get a gradeable task instead of an anecdote.

07

What a loose spec costs

Now delete Program.cs from your head for a moment and imagine you had only asked for “a class that sorts an array of doubles from low to high”. A perfectly reasonable implementation of that sentence is: 

public static double[] Ascending(double[] values)
{
    Array.Sort(values);
    return values;
}

Two lines, sorts low to high, and it is correct against the English. It also passes rung 1, passes rung 2, builds clean, and returns exactly the values you asked for. Every oracle in the harness that does not know what you meant says yes: 

  pass  every value is returned (expected 5, got 5)
  pass  the smallest value comes first (expected -1.25, got -1.25)
  pass  the largest value comes last (expected 10, got 10)
  FAIL  the input array is not reordered (expected 3.5, got -1.25)
  pass  an empty array sorts to an empty array (expected 0, got 0)
  pass  NaN sorts before every number
  pass  a null input throws ArgumentNullException
1 TEST(S) FAILED

It sorted the caller's array in place. Somewhere upstream, a variable the caller still believed in has quietly changed. No compiler can see that, no diff review would flag it — the diff looks great — and the agent had no way to know, because nobody told it.

The general shape

The verification ladder gets cheaper and stronger as you climb: syntax, compile, analyzers, tests. Everything below the test rung is checking that the code is well-formed. Only the test rung checks that it is right, and only against the properties you wrote down. An agent with a compiler and no tests is an agent that can be confidently, buildably wrong — which is why the four checks in lesson 1 were worth the ten minutes they took.

08

Exercises

TryWhat you should observe
Run the same goal twice without deleting the file The second run gets already_exists and has to switch verbs. Watch whether it reaches for edit_file on its own — the hint tells it to, and whether a model takes that hint is worth knowing about your model.
Add Descending to the spec and ask for both Now the second method has to go into a file that already exists, so the run needs both verbs. Two near-identical bodies is also where The text to replace appears 2 times in 'DoubleSorter.cs'. starts showing up.
Set Verification:RejectEditsThatBreakTheBuild to false Lesson 4's refusal becomes a report. The file with the hallucinated API lands on disk, and build is left to catch it a step later.
Empty WritablePaths and rerun path_not_allowed: No writable paths are configured, so every write is rejected. Creation is gated by the same allow-list as editing — read-only is enforced by the workspace, not by asking the model nicely.
Press Reject on a good class Row turns red with A reviewer rejected this change. The loop resumes, the agent is told the write was refused, and what it tries next is the interesting part.
Change the spec so NaN must sort last The natural implementation now fails. This is the cheapest demonstration that the spec, not the goal text, is what the agent is really being measured against.
Ask for the class with no Program.cs at all It will write something plausible, the build will pass, and you will have no way to tell whether it is right. That is the state most ad-hoc agent runs are actually in.
Run the same goal twice and group Metrics by Task Both land under desktop, and pass@1 shows - rather than a percentage: ad-hoc goals are never graded against an oracle. Only suite tasks are.
