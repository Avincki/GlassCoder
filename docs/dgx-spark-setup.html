01

The binding constraint

A DGX Spark holds 128 GB of unified LPDDR5x that the CPU and GPU share, at roughly 273 GB/s. For comparison, an RTX PRO 6000 moves memory around five times faster. So the Spark is not a small machine that runs small models — it is a capacious machine that reads memory slowly.

Autoregressive decoding reads the active weights once per generated token. That gives you a ceiling you can work out on the back of an envelope: 

tokens/second  ≈  273 GB/s  ÷  bytes read per token

70B dense, 4-bit  → ~35 GB/token  →  ~7 tok/s   (unusable for an agent)
120B MoE, 12B active, NVFP4 → ~6 GB/token → ~23 tok/s  (workable)
80B MoE, 3B active,  4-bit  → ~2 GB/token → fast, but a smaller brain

The conclusion is unusually clean, and it is the opposite of the advice you would give for a bandwidth-rich GPU: on this box, pick a sparse mixture-of-experts model, not a dense one. An MoE reads only its active experts per token, so it buys you a large parameter count at a small bandwidth cost. vLLM's own guidance for the Spark is to look for 100–130B total parameters with 10–15B active, in NVFP4.

Why this matters for an agent specifically

GlassCoder measures wall-clock per solved task alongside pass@1 for exactly this reason. A thirty-step run at 23 tok/s is minutes of decoding; the same run on a dense 70B at 7 tok/s is a coffee break. The model that scores better on a benchmark can still be the wrong model for the loop.

What actually fits

NVFP4 weights are roughly half a byte per parameter. After the model, you need room for the KV cache, the container runtime and the OS — vLLM's own recipe leaves ~15% headroom.

Nemotron-3-Super-120B-A12B, NVFP4 ~66 GB weights · 128 GB total
weights ~66 GB KV cache headroom
Qwen3.5-122B-A10B, NVFP4 ~76 GB weights · 128 GB total
weights ~76 GB KV cache headroom
A dense 70B at 8-bit ~70 GB weights — fits, but reads all of it every token
weights ~70 GB KV cache headroom
model weights KV cache at 128K context OS, runtime, growth

Note that the dense 70B fits perfectly well. Fitting was never the problem.

02

Picking the model

Named models age fast — treat the table below as a snapshot taken in July 2026 and the criteria under it as the part that lasts.

ModelTotal / activeFit on one SparkWhy
NVIDIA Nemotron-3-Super NVFP4
start here		 120B / 12B good fit vLLM's own reference recipe for this box, with published numbers: ~23 tok/s decode. Ships as NVFP4, so no quantising yourself.
Qwen3.5-122B-A10B NVFP4 122B / 10B good fit ~76 GB quantised, leaving ~52 GB for cache. Community-quantised specifically for the Spark.
Qwen3-Coder-Next 80B / 3B verify first Very few active parameters, so potentially fast, and coding-tuned. Reported throughput varies a lot by quantisation and runtime — measure before you commit to it.
GLM-5.2 very large needs ~4 Sparks Currently the strongest open-weight coding scores, and out of reach of a single 128 GB box. Relevant only if you cluster.
Any dense 70B+ 70B / 70B too slow Fits in memory, reads all of it per token. Around 7 tok/s — the loop will feel broken even when it is working.

The criteria that outlive the table

  1. Tool-call fidelity first. GlassCoder is a tool-calling harness; a model that writes beautiful code but emits malformed tool calls is worthless to it. Check that vLLM has a --tool-call-parser matching the model's chat template before you download 70 GB.
  2. Sparse, not dense. 10–15B active parameters is the target. Below that you lose capability; above it you lose the wall-clock.
  3. NVFP4 checkpoints where they exist. Blackwell has native FP4 kernels. Prefer a pre-quantised NVFP4 release over quantising a BF16 checkpoint yourself.
  4. Long context, but do not pay for it twice. 128K served is plenty; GlassCoder budgets ~48K and compacts, so you are not sizing the KV cache for the worst case.
03

Standing up vLLM

Check this before anything else

The Spark is arm64 with sm_121, and upstream stable vLLM has not always built for it. Use NVIDIA's container or the CUDA-13 track, not the default vllm/vllm-openai:latest. This is the single most common way to lose an afternoon here.

  1. Confirm the box

    nvidia-smi                       # GB10, driver, CUDA 13
uname -m                         # aarch64
free -g                          # ~128 GB unified
docker info | grep -i runtime    # nvidia runtime present

  2. Pull a container that knows about GB10

    Either NVIDIA's NGC build or vLLM's CUDA-13 track. Pin the tag — "nightly" is a compatibility track, not a reproducible one, and you want the ability to say which build produced a given metrics row. 

    # NVIDIA's build (recommended starting point)
docker pull nvcr.io/nvidia/vllm:26.02-py3

# or vLLM's CUDA-13 track
docker pull vllm/vllm-openai:cu130-nightly

    NVIDIA also publishes step-by-step recipes for this hardware in the dgx-spark-playbooks repository, including a vLLM playbook. Start from there if the container route fights you.

  3. Serve the worker

    Every flag below is load-bearing on this hardware. The two that matter most to GlassCoder are the served name and the tool-call parser. 

    docker run -d --name vllm-worker --restart unless-stopped \
  --gpus all --ipc=host -p 8001:8000 \
  -e HF_TOKEN="$HF_TOKEN" \
  -v ~/.cache/huggingface:/root/.cache/huggingface \
  nvcr.io/nvidia/vllm:26.02-py3 \
  vllm serve nvidia/NVIDIA-Nemotron-3-Super-120B-A12B-NVFP4 \
    --served-model-name worker \
    --max-model-len 131072 \
    --gpu-memory-utilization 0.85 \
    --max-num-seqs 4 \
    --enable-auto-tool-choice \
    --tool-call-parser qwen3_coder \
    --guided-decoding-backend xgrammar \
    --trust-remote-code

  4. Wait, then warm it

    First load reads tens of gigabytes from disk — expect 10–15 minutes. The first request after that triggers kernel compilation, around 25 seconds. Fire a throwaway request so a real one never pays for it. 

    curl -s localhost:8001/v1/chat/completions \
  -H 'content-type: application/json' \
  -d '{"model":"worker","messages":[{"role":"user","content":"hi"}],"max_tokens":3}'

What each flag is doing

--served-model-name
The alias clients address. GlassCoder requires an alias and rejects anything that looks like a checkpoint path at startup, so name it worker and keep serving topology out of your configuration.
--gpu-memory-utilization
0.85 on a unified pool. This memory is also the operating system's — take it all and you will meet the OOM killer rather than a clean error.
--max-num-seqs
4. Above roughly four concurrent decode streams the per-token bandwidth tax outweighs what continuous batching gives back. This is a bandwidth limit, not a compute one.
--enable-auto-tool-choice
Without this, tool calls come back as prose in the message body and every one of them is an invalid call.
--tool-call-parser
Must match the model's chat template — qwen3_coder, hermes, llama3_json and others. Wrong parser looks exactly like a stupid model.
--guided-decoding-backend
The grammar engine behind constrained decoding. GlassCoder leans on this hard: it is what makes a malformed tool call structurally impossible rather than merely unlikely.
--reasoning-parser
Only for models that emit reasoning blocks, and only the parser that model documents. Omit it otherwise.
04

Serving the three roles

GlassCoder addresses three served aliases: worker drives the loop, drafter is there for a larger model on hard sub-problems, and critic runs the refutation pass. You do not need three models to start.

One Spark, one model

Point all three roles at the same endpoint and alias. Phases 0 and 1 only ever use worker, so this costs you nothing until Phase 2.

The caveat to write down

When the critic is the same model as the worker, its blind spots are the worker's blind spots — it will happily approve the reasoning it just produced. That is not a fatal problem, but it means recovery rate from a same-model critic is an optimistic number. Note it next to the metric, or the Phase 2 result will read better than it is.

One Spark, two models

A better arrangement once you reach Phase 2: keep the big MoE as the worker, and put a small dense model from a different family on a second port as the critic. Refutation is a much easier job than authoring, so a 7–14B model does it well and costs little memory. 

docker run -d --name vllm-critic --gpus all --ipc=host -p 8003:8000 \
  -v ~/.cache/huggingface:/root/.cache/huggingface \
  nvcr.io/nvidia/vllm:26.02-py3 \
  vllm serve <a-small-model-from-another-family> \
    --served-model-name critic \
    --max-model-len 32768 \
    --gpu-memory-utilization 0.10 \
    --max-num-seqs 4

The two utilisation fractions must add up to something below 1.0 with room to spare — they are drawing on the same pool. Drop the worker to 0.75 if you do this.

05

Wiring GlassCoder to it

In config/appsettings.json, the roles point at the ports you just opened. Nothing else in the harness needs to know what hardware this is. 

"Models": {
  "DefaultRole": "worker",
  "Roles": {
    "worker": {
      "Endpoint": "http://spark.local:8001/v1",
      "ModelAlias": "worker",
      "TimeoutSeconds": 900,
      "Temperature": 0.2,
      "ConstrainedDecoding": {
        "Enabled": true,
        "StrictToolSchemas": true,
        "SingleToolCallPerStep": true,
        "GuidedDecodingBackend": "xgrammar"
      }
    },
    "critic": {
      "Endpoint": "http://spark.local:8003/v1",
      "ModelAlias": "critic",
      "Temperature": 0.0
    }
  }
}

Settings worth changing for this hardware

TimeoutSeconds
Raise to 900. A long prefill plus a slow decode can legitimately exceed the 600 default on a large prompt.
Agent:MaxWallClockSeconds
At ~23 tok/s, a thirty-step run is genuinely minutes. Budget 1800 rather than 900, or you will record time-limit stops that are really just honest slowness.
Orchestration:
MaxDegreeOfParallelism
Keep at 3, below the server's --max-num-seqs 4. Fanning out wider does not go faster on a bandwidth-bound box — it just makes every sub-agent slower.
Context:
MaxContextTokens
48000 against 131072 served is a deliberate gap. Prefill is charged on every step, so a bloated window costs wall-clock on every turn, not just the one that grew it.
Cost per million
tokens
Leave at 0 for a local arm, and compare on wall-clock instead. If you want a real figure, divide the box's power draw and amortised cost by tokens produced and put that in — then a local arm and a hosted arm become comparable.
06

Verifying it properly

"The endpoint returns text" is a much weaker claim than "the endpoint calls tools correctly". Check the second one, because that is what the harness needs.

  1. The endpoint answers

    curl -s localhost:8001/v1/models | jq '.data[].id'   # → "worker"

  2. It emits a real tool call

    The response must contain a tool_calls array. If the function name and arguments appear as text inside content instead, your parser flag is wrong — fix that now, not after a confusing run. 

    curl -s localhost:8001/v1/chat/completions \
  -H 'content-type: application/json' -d '{
  "model": "worker",
  "messages": [{"role":"user","content":"Read the file src/Program.cs"}],
  "tools": [{"type":"function","function":{
    "name":"read_file",
    "description":"Read a text file from the workspace.",
    "parameters":{"type":"object","properties":{
      "path":{"type":"string","description":"Repo-relative path."}},
      "required":["path"]}}}],
  "tool_choice": "auto"
}' | jq '.choices[0].message.tool_calls'

  3. GlassCoder resolves against it

    glasscoder tools                     # config binds, services resolve
glasscoder run --repo /path/to/repo \
  --goal "Find where retries are implemented and explain the backoff."

  4. Read the number that decides everything

    Tool-call validity rate is the acceptance test for this whole setup. Near 1.0 means the seam is healthy and you can start measuring the harness. Anything much lower is a server or parser problem, not a model problem — go back to step 2. 

    jq -r '"\(.taskId)  validity=\(.toolCallValidityRate)  steps=\(.steps)"' \
  metrics/metrics.jsonl | tail -5

  5. Take the Phase 0 baseline

    Read-only, so nothing can be damaged, and it gives you the number every later change is measured against. 

    glasscoder suite --config config/phase0.json
07

Budgets that now make sense

Published figures for a 120B/12B NVFP4 model on a single Spark, to calibrate what "slow" and "broken" look like:

MeasureObservedWhat it implies
Decode ~23 tok/s A 500-token step takes ~22s. Ten steps is about four minutes of pure decoding.
Prefill, short prompt ~140 tok/s Small prompts are dominated by fixed overhead.
Prefill, long prompt ~1,884 tok/s Prefill parallelises well — it is decode that is bandwidth-bound.
Time to first token 0.4 – 3.9 s Scales with prompt length. This is why context compaction pays on every step.
The measurement this hardware makes interesting

Because prefill is fast and decode is slow, anything that shortens the output is worth more here than anything that shortens the input. That is a real, testable hypothesis — and it is exactly what the ablation runner exists for. Run no-context against baseline and see whether the context you are loading actually buys enough to justify itself.

08

If one Spark isn't enough

Two Sparks connect directly over their 200GbE ConnectX-7 ports, giving a combined 256 GB pool; larger clusters go through a switch. People report running 300B-class MoE models across four units, and current flagship open-weight coding models need roughly that.

Before you buy a second box, be honest about which problem you have:

Model too big
More Sparks genuinely help — you are buying memory capacity.
Runs feel slow
More Sparks help far less than you would hope. Aggregate bandwidth rises but so does interconnect latency, and decode stays the bottleneck. Try a smaller active parameter count first.
Want a real critic
A second small model on the same box is usually the better spend than a second box.

Whatever you choose, keep the aliases stable. If a second machine serves drafter, GlassCoder needs one line of configuration and no code change — which is the entire point of the seam.

09

Known rough edges

SymptomCause and fix
Container exits immediately, CUDA arch error Stable upstream vLLM built without sm_121. Use NVIDIA's NGC image or the CUDA-13 track, or follow the DGX Spark playbook to build with sm_120 forward compatibility.
Tool-call validity near zero Almost always --enable-auto-tool-choice missing, or a --tool-call-parser that does not match the chat template. Verify with the raw curl above before suspecting the model.
First GlassCoder run times out Model load is 10–15 minutes and first-request JIT is ~25s. Warm the server before pointing the harness at it.
OOM killer, or the desktop stalls --gpu-memory-utilization too high. This pool is shared with the operating system. Back off to 0.80 and re-measure.
Throughput collapses under load Too many concurrent sequences for the bandwidth. Lower --max-num-seqs, and keep GlassCoder's fan-out below it.
Startup takes forever every time Default safetensors loading. Look at faster loaders, and keep the HuggingFace cache on the internal NVMe rather than a network mount.
Quality drops after long runs If you enabled --kv-cache-dtype fp8 to save memory, turn it back off — it trades predictability for space. Prefer compaction, which GlassCoder already does.
Dated material

Everything above was checked in July 2026. Container tags, sm_121 support and the model landscape all move quickly; the arithmetic in section 1 does not. When a recommendation here has gone stale, re-derive from bandwidth ÷ bytes-per-token and you will land somewhere sensible.
